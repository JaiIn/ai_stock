## Purpose

MS-09.01 candidate input contract. It intentionally does not implement
recommendation, scoring, persistence, file loading, UI, DB access, Toss API,
OpenAI/LLM, OAuth, accountSeq, account, order, balance, asset, or fill behavior.

## Watchlist Data Model Summary

The model accepts caller-supplied watchlist items and collections, validates
them deterministically, and can convert watchlist items into MS-09.01
`CandidateInput` objects. It does not load from files, store to files, query a
database, write a database, score items, rank items, recommend items, trade, or
render UI.

## Validation Policy Summary

- Watchlist name is trimmed and empty name returns `invalid_watchlist_name`.
- Empty item collection returns `empty_watchlist`.
- Item symbol, source, disabled state, duplicates, and data hints are validated
  through the MS-09.01 candidate input contract.
- Priority is optional and must remain within the policy range when supplied.
- Tags are normalized as trimmed non-empty strings.
- Group, note, and reason are caller-supplied safe text only.

## Duplicate Handling Policy

Duplicates are marked deterministically by normalized symbol within the
caller-supplied watchlist. The model reports duplicate status only; it does not
store, merge, rewrite, or de-duplicate a persisted watchlist.

## Disabled Item Handling Policy

Disabled items receive the safe `disabled_candidate` item status and are not
selectable. A collection with disabled items is reported as
`contains_disabled_candidates`.

## Insufficient Data Handling Policy

Partial or missing data hints are carried through as safe `insufficient_data`
or `needs_review` states. These states are not recommendations and do not
produce scoring or action labels.
