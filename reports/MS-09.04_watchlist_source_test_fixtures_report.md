## Watchlist Source Fixture Summary

The fixture module defines in-memory fixture records and an expected-vs-actual
evaluation result. Each fixture supplies a `WatchlistSourceConfig`; actual
watchlist/candidate output is produced only through the existing MS-09.03
`build_watchlist_source_result` adapter. The fixture layer does not load,
persist, score, recommend, trade, render UI, read DB rows, read files, write
files, or call external endpoints.

## Contract Reuse

- MS-09.01 candidate input contract is reused through the MS-09.03 adapter.
- MS-09.02 watchlist model is reused through the MS-09.03 adapter.
- MS-09.03 manual/local source adapter is the only path used to produce actual
  watchlist source results from fixture source configs.
- Allowed source, forbidden source, forbidden field, forbidden label, and
  required false flag policies are reused rather than reimplemented as a
  storage, UI, scoring, or API path.

## Expected vs Actual Evaluation Policy

`evaluate_watchlist_fixture` builds the actual source result through
`build_watchlist_source_result` and compares the actual watchlist status,
candidate statuses, required false flags, expected rejection keywords, and
expected diagnostics keywords against the fixture record. Failures are returned
as deterministic strings; no exception, I/O, DB lookup, file lookup, API call,
scoring, or UI rendering is required.

## Fixture Behavior Summaries

- Forbidden field sanitization fixture confirms forbidden caller fields are
  reported as safe diagnostics/rejection reasons and forbidden values are not
  copied into watchlist or candidate output models.
- Forbidden label/action prevention is verified by ensuring buy/sell/hold and
  related labels are not produced as action labels, recommendation results,
  scoring outputs, or execution directives.
- Duplicate fixture confirms duplicate symbols produce `duplicate_candidate`.
- Disabled fixture confirms disabled items produce `disabled_candidate`.
- Insufficient data fixture confirms partial data produces `insufficient_data`
  and a safe `needs_review` watchlist status.
- Empty watchlist fixture confirms empty caller-supplied records produce
  `empty_watchlist` safely.
