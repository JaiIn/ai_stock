# MS-03.02 Data Persistence Service Report

## 수행 내용

- `LocalDataPersistenceService`를 추가해 이미 파싱된 모델 객체를 로컬 SQLite repository에 저장하는 얇은 서비스 계층을 구현했다.
- 서비스는 `StockRepository`, `MarketDataRepository`, `MarketInfoRepository`만 사용한다.
- 실제 Toss API client, OAuth token provider, authenticated request context, httpx transport는 import하거나 생성하지 않았다.
- 테스트는 `:memory:` SQLite DB만 사용했다.

## 서비스 구조

- `save_stocks(stocks)`: `StockInfo` 목록을 upsert 방식으로 저장한다.
- `save_price_snapshots(snapshots)`: `PriceSnapshot` 목록을 append insert 방식으로 저장한다.
- `save_candles(stock_code, interval, candles)`: `Candle` 목록을 append insert 방식으로 저장한다.
- `save_exchange_rates(rates)`: `ExchangeRate` 목록을 append insert 방식으로 저장한다.
- 조회 helper는 저장 검증과 후속 service 연동을 위해 최소 범위로 제공한다.

## Repository 연동

- 생성자에서 SQLite connection을 받아 schema를 초기화하고 repository 인스턴스를 구성한다.
- 서비스 입력은 이미 생성된 domain model 객체로 제한했다.
- 외부 API 응답 원문, 인증정보, 계좌정보는 저장하거나 다루지 않는다.

## 로컬 전용 및 실제 호출 없음 확인

- 모든 테스트는 `create_connection(":memory:")` 기반으로 실행했다.
- workspace에 `data/ai_stock.sqlite3` 또는 기타 `*.sqlite`, `*.sqlite3`, `*.db` 파일을 생성하지 않았다.
- 테스트에서 network socket을 patch해 service 조회 중 실제 네트워크 호출이 발생하지 않음을 확인했다.

## 민감정보 저장 없음 확인

- 서비스와 테스트는 실제 Client ID, Client Secret, Access Token, accountSeq, 계좌번호를 요구하거나 저장하지 않는다.
- API key/token/secret 저장 테이블이나 관련 서비스는 추가하지 않았다.
- 계좌/자산/주문/실거래/포트폴리오/잔고/체결 서비스는 구현하지 않았다.

## 추후 보완 사항

- 중복 key 정책은 후속 Micro Stage에서 확정한다.
- retention policy는 후속 Micro Stage에서 확정한다.
- timestamp/date_time 정규화와 timezone 정책은 후속 Micro Stage에서 확정한다.
- 파일 기반 로컬 DB 초기화/운영 흐름은 사용자 승인된 별도 Micro Stage에서만 검토한다.
