# MS-04.02 Paper Order Validation Service Report

## 수행 내용

- `PaperOrderValidationService`를 추가해 `PaperOrder`가 `PaperPortfolio` 기준으로 유효한지 검증하는 read-only 서비스를 구현했다.
- 검증 결과 모델 `PaperOrderValidationResult`와 예외 `PaperOrderValidationError`를 추가했다.
- 기존 `assert_paper_trading_only()` safety guard를 재사용했다.
- 실제 주문 전송, 실제 체결, 실제 계좌 조회, repository/SQLite 저장소는 구현하지 않았다.

## 검증 규칙

### 매수

- 기존 `PaperOrder` 모델 검증에 따라 수량과 가격은 0보다 커야 한다.
- 예상 주문 금액은 `quantity * price`로 `Decimal` 계산한다.
- 예상 주문 금액이 모의 현금 잔고보다 크면 거절한다.
- 수수료/세금은 이번 단계에서 0으로 간주하며 후속 Micro Stage에서 별도 모델링한다.

### 매도

- 기존 `PaperOrder` 모델 검증에 따라 수량과 가격은 0보다 커야 한다.
- 해당 종목의 모의 보유가 없으면 거절한다.
- 보유 수량보다 많이 매도하려 하면 거절한다.

### 주문 상태

- `CREATED` 상태의 paper order만 검증 대상으로 허용한다.
- `FILLED`, `CANCELLED`, `REJECTED` 등 terminal 상태의 order는 거절한다.

## Safety guard 사용

- `assert_paper_trading_only()`를 검증 시작 시 호출한다.
- live/real/broker/production 모드 또는 real-order flag가 들어오면 `PaperTradingSafetyError`가 발생한다.

- `socket.socket` patch로 테스트 중 네트워크 호출이 발생하지 않음을 확인했다.
- Toss client, OAuth, httpx Client, authenticated request context, send 호출을 사용하지 않았다.
- 실제 주문 API, 실계좌 주문 전송, 계좌/자산/잔고/체결 API는 구현하지 않았다.
- 포트폴리오 cash/holding 변경 로직, 본격 체결 엔진, 본격 수익률 계산 엔진은 구현하지 않았다.
- repository, SQLite schema, DB 파일, pyproject는 변경하지 않았다.

- 체결 엔진과 포트폴리오 업데이트는 별도 Micro Stage에서 구현한다.
