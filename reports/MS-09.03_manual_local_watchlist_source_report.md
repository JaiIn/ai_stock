# MS-09.03 Manual/Local Watchlist Source Report

## Purpose

MS-09.03 adds a pure no-I/O manual/local source adapter on top of the
MS-09.01 candidate input contract and MS-09.02 watchlist data model. It accepts
only caller-supplied symbols or item dictionaries, normalizes them into
watchlist items, validates the resulting watchlist, and returns candidate
inputs for later stages.

The stage intentionally does not implement recommendation, scoring, watchlist
persistence, file loading, file reading/writing, UI, DB access, Toss API,
OpenAI/LLM, OAuth, accountSeq, account, order, balance, asset, or fill behavior.

## Manual/Local Watchlist Source Summary

The adapter converts caller-supplied local/manual source records into
`Watchlist` and `WatchlistItem` objects, validates them through the MS-09.02
watchlist model, and exposes MS-09.01 `CandidateInput` objects through a pure
conversion. It does not load from paths, query databases, call APIs, request
credentials, score items, rank items, recommend items, trade, or render UI.

## Contract Reuse

- MS-09.01 candidate input contract is reused for candidate inputs, allowed and
  forbidden candidate sources, candidate statuses, and forbidden label policy.
- MS-09.02 watchlist model is reused for watchlist items, collections,
  validation, summaries, duplicate handling, disabled item handling,
  insufficient-data handling, and candidate conversion.

## Manual Symbol List Policy

- Caller-supplied symbol strings are trimmed and converted to `WatchlistItem`
  objects.
- Empty symbols are rejected by the reused candidate validation contract.
- Duplicate symbols are marked deterministically as `duplicate_candidate`.
- Default market, tags, group, reason, and enabled state are applied.
- Unknown market is normalized safely by the reused candidate contract.
- No storage, file loader, DB lookup, API lookup, scoring, or UI behavior is
  performed.

## Manual Item Dictionary Policy

- Caller-supplied dictionaries are converted using only allowed fields:
  `symbol`, `name`, `market`, `source`, `enabled`, `reason`, `tags`, `group`,
  `priority`, `note`, and `data_availability_hint`.
- Missing source falls back to the safe source mapped from source type.
- Forbidden candidate source values are rejected by validation.
- Disabled items receive `disabled_candidate`.
- Priority, tags, group, note, reason, and data hints are normalized.
- Forbidden fields are not copied into the output `WatchlistItem` or
  `CandidateInput`.

## Forbidden Field Detection Policy

Forbidden keys are detected from caller-supplied dictionaries and reported as
safe diagnostics/rejection reasons without including their values in output
models.

Detection terms:

```text

authorization
api_key

client_secret

balance

```

These terms are policy/test/documentation terms only and are not emitted as
recommendation, action, scoring, or execution labels.

## Duplicate Handling Policy

Duplicates are marked deterministically by normalized symbol inside the
caller-supplied source result. The adapter reports duplicate status only; it
does not persist, merge, rewrite, or de-duplicate stored watchlists.

## Disabled Item Handling Policy

Disabled items are kept as safe non-selectable items with
`disabled_candidate` validation status. They do not trigger any UI, order,
account, or recommendation behavior.

## Insufficient Data Handling Policy

Insufficient or partial data hints are carried through as safe
`insufficient_data` or `needs_review` states via the reused watchlist model.
These states are not recommendations and do not produce scoring or action
labels.
