## endpoint별 재검증 결과 요약

| Category | Endpoint | 재검증 결과 |
|---|---|---|
| Auth / OAuth | `POST /oauth2/token` | Form body 기반 OAuth token endpoint입니다. Response는 OAuth 표준 field를 사용하며 일반 `result` envelope가 아닙니다. |
| Stock Info | `GET /api/v1/stocks` | Query `symbols`는 comma-separated, max 200입니다. Response result는 stock info array입니다. |
| Stock Info | `GET /api/v1/stocks/{symbol}/warnings` | Path `symbol`을 사용하며 response result는 warning array입니다. |
| Market Data | `GET /api/v1/orderbook` | Query `symbol`; result는 orderbook object입니다. 현재 parser/model은 후속 구현 대상입니다. |
| Market Data | `GET /api/v1/prices` | Query `symbols`; result는 price snapshot array입니다. |
| Market Data | `GET /api/v1/trades` | Query `symbol`, optional `count`; result는 trade array입니다. |
| Market Data | `GET /api/v1/price-limits` | Query `symbol`; result는 upper/lower limit price object입니다. |
| Market Data | `GET /api/v1/candles` | Query `symbol`, `interval`, optional pagination fields; result root는 `candles`와 `nextBefore`를 포함하는 object입니다. |
| Market Info | `GET /api/v1/exchange-rate` | Query는 optional `dateTime`만 확인되었습니다. Result는 `baseCurrency`, `quoteCurrency`, `rate`, `validFrom`, `validUntil`입니다. |
| Account / Asset / Order Info | 여러 GET endpoint | Read-only 후보이나 accountSeq 또는 계좌 정보를 다루므로 별도 승인 전까지 구현 보류입니다. |
| Order Mutation | 여러 POST endpoint | 실제 주문 생성/정정/취소 계열이므로 denylist 유지 대상입니다. |

## 프로젝트 기존 mock client 가정과 공식 문서 간 차이

1. Auth token endpoint
   - 기존 token model/mock provider 구조는 방향이 맞습니다.
   - Token response는 일반 `result` envelope가 아니라 OAuth 표준 response입니다.

2. Stock Info
   - `getStocks`, `getStockWarnings`의 path/query/response root 가정은 대체로 일치합니다.
   - 현재 모델은 공식 optional field 전체를 반영하지 않았습니다.

3. Market Data
   - `getPrices`는 path/query/response root 가정이 대체로 일치합니다.
   - `getCandles`는 공식 response root가 object이며 `candles`, `nextBefore`를 포함합니다. 현재 parser 가정은 후속 alignment가 필요합니다.
   - `orderbook`, `trades`, `price-limits`는 request definition 중심이고 parser/model은 후속 범위입니다.

4. Market Info / Exchange Rate
   - 현재 mock client는 `baseCurrency`, `quoteCurrency`, `dateTime` query를 가정합니다.
   - 공식 schema에서는 optional `dateTime`만 query로 확인되었습니다.
   - 현재 model/parser는 `exchangeRate`, `dateTime`을 가정하지만 공식 response는 `rate`, `validFrom`, `validUntil`을 사용합니다.
   - MS-05.02 이후 가장 우선적인 code alignment 후보입니다.

## schema confidence level

| Scope | Confidence | 근거 |
|---|---|---|
| Auth / OAuth path/method/body/response | High | 공식 OpenAPI에서 확인 |
| Stock Info path/query/response root | High | 공식 OpenAPI에서 확인 |
| Market Data prices path/query/response root | High | 공식 OpenAPI에서 확인 |
| Market Data candles current parser compatibility | Medium | 공식 schema와 현재 parser 가정 차이 확인 |
| Market Info exchange-rate current compatibility | Medium | 공식 schema와 현재 query/model/parser 가정 차이 확인 |
| Account/Asset/Order read-only future support | Low | 구현 범위 밖이며 accountSeq 안전 정책 재검토 필요 |
| Order Mutation denylist | High | 공식 OpenAPI에서 write/order mutation endpoint 확인 |

## 불확실하거나 후속 확인 필요한 항목

- `getExchangeRate` query/model/parser 공식 schema alignment
- `getCandles` parser의 `result.candles` / `nextBefore` 처리
- `orderbook`, `trades`, `price-limits` 최소 response model/parser 구현 여부
- `StockInfo` optional official field 확장 범위
- Account/Asset/Order History/Order Info read-only API의 안전 정책과 accountSeq 취급 방식
- numeric field를 Decimal로 복원하는 범위와 timestamp/dateTime 정규화 정책
